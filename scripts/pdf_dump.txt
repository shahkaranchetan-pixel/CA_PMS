2.1  Inconsistent Authentication Pattern 
Some routes check authentication, others don't. This is the root cause of your security issues above. A consistent 
pattern is needed across all API routes. 
 
Current (Inconsistent) Pattern 
// Some routes: no auth check at all 
export async function GET() { 
  const clients = await prisma.client.findMany() // Unprotected! 
} 
 
Recommended: Create a reusable auth helper — src/lib/auth-helpers.ts 
import { getServerSession } from 'next-auth' 
import { authOptions } from '@/app/api/auth/[...nextauth]/route' 
import { NextResponse } from 'next/server'

export async function requireAuth(role?: 'ADMIN' | 'EMPLOYEE') { 
  const session = await getServerSession(authOptions) 
  const user = session?.user as any 
  if (!user) return { error: NextResponse.json({ error: 'Unauthorized' }, { status: 401 }) } 
  if (role && user.role !== role) return { error: NextResponse.json({ error: 'Forbidden' }, { 
status: 403 }) } 
  return { user } 
} 
 
// Usage in every API route: 
export async function GET() { 
  const { user, error } = await requireAuth() 
  if (error) return error 
  // ... rest of handler 
} 
 
2.2  Sequential DB Calls That Should Be Parallel 
In page.tsx (Dashboard), multiple database queries are run one after another with await. These are independent 
queries and should run in parallel with Promise.all(). 
 
Current (Slow) Pattern in src/app/page.tsx 
const totalClients = await prisma.client.count()          // Wait... 
const activeClients = await prisma.client.findMany(...)   // Then wait... 
const allActiveTasks = await prisma.task.findMany(...)    // Then wait... 
const completedThisMonth = await prisma.task.count(...)   // Then wait... 
 
Recommended: Parallel queries with Promise.all() 
const [totalClients, activeClients, allActiveTasks, completedThisMonth, recentLogs] = 
  await Promise.all([ 
    prisma.client.count(), 
    prisma.client.findMany({ where: { active: true }, select: { name: true, id: true } }), 
    prisma.task.findMany({ where: { ...baseTaskWhere, status: { not: 'COMPLETED' } }, ... }), 
    prisma.task.count({ where: { ...baseTaskWhere, status: 'COMPLETED', ... } }), 
    prisma.taskLog.findMany({ take: 8, orderBy: { createdAt: 'desc' }, ... }), 
  ]) 
This will reduce dashboard load time significantly — from ~5 sequential DB round-trips to 1 parallel batch.

2.4  Role Stored as String — Should Be Enum 
The User model stores role as a plain String with a comment // ADMIN, EMPLOYEE. If someone accidentally 
types 'Admin' or 'admin' it won't match your role checks. Use a Prisma enum. 
 
// In schema.prisma — change this: 
role  String  @default("EMPLOYEE") // ADMIN, EMPLOYEE 
 
// To this: 
enum Role { ADMIN EMPLOYEE } 
role  Role  @default(EMPLOYEE) 
 
2.5  Mailer Creates a New Transporter on Every Email 
In mailer.ts, a new nodemailer.createTransport() is called every time sendEmail() is invoked. For bulk reminders 
(50+ clients), this creates 50+ transporter instances. 
 
// Recommended: Create a singleton transporter 
let _transporter: nodemailer.Transporter | null = null 
 
async function getTransporter() { 
  if (_transporter) return _transporter 
  const config = await getSmtpConfig() 
  _transporter = nodemailer.createTransport({ host: config.SMTP_HOST, ... }) 
  return _transporter 
} 
 
2.6  Dashboard Over-fetches Data 
The dashboard fetches ALL active tasks with full includes (client, assignees) to then filter them in JavaScript. This 
means pulling potentially thousands of records from the DB when only counts and limited data are needed. 
• allActiveTasks includes full client and assignee objects, then .filter() is used for overdue/blocked tasks 
• team query includes all taskAssignees with full task objects for every user 
• Solution: Use Prisma aggregations (count, groupBy) and targeted queries instead of fetching all records

04. Recurring Task Auto-Rollover 
  Monthly compliance tasks should appear automatically — 
zero manual effort
 
Today your admin must click 'Generate Monthly Tasks'. This should be invisible — tasks should 
just appear. 
 
Your app has a MONTHLY frequency field on tasks but it doesn't actually auto-create the next month's task when 
the current one is completed. Here is how to close that gap: 
 
Approach: Trigger on Completion 
When a task with frequency=MONTHLY is marked COMPLETED, automatically create the next month's instance. 
This is the most reliable approach as it ties task creation to actual completion, preventing duplicate creation. 
 
// Add to PATCH /api/tasks/[id]/route.ts 
// After successfully updating status to COMPLETED: 
 
if (status === 'COMPLETED' && oldTask.frequency === 'MONTHLY') { 
  const [mon, yr] = (oldTask.period || '').split('-') 
  const monthIndex = MONTHS[mon] 
  const nextMonth = new Date(parseInt(yr), monthIndex + 1, 1) 
  const nextPeriod = `${MONTH_NAMES[nextMonth.getMonth()]}-${nextMonth.getFullYear()}` 
 
  // Check if next month's task already exists 
  const exists = await prisma.task.findFirst({ 
    where: { clientId: oldTask.clientId, taskType: oldTask.taskType, period: nextPeriod } 
  }) 
 
  if (!exists) { 
    await prisma.task.create({ 
      data: { 
        title: oldTask.title.replace(oldTask.period!, nextPeriod), 
        taskType: oldTask.taskType, 
        frequency: 'MONTHLY', 
        period: nextPeriod, 
        clientId: oldTask.clientId, 
        dueDate: calculateDueDate(oldTask.taskType, nextMonth), 
        status: 'PENDING', 
      } 
    }) 
  } 
}
